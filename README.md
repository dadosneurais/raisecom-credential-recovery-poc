# Raisecom Gateway - Broken Authentication & Weak Credential Obfuscation

### High-level overview:
The vulnerability lies in the improper implementation of credential protection during the authentication process. The device uses a reversible XOR-based obfuscation method where the "secret" key is generated on the client-side and sent alongside the obfuscated password. This allows an attacker to perform offline brute-force attacks and pre-calculate "rainbow tables" for any given password dictionary, bypassing the intended security entropy.
---

### Extended Impact & Threats:
* Offline Brute-Force Acceleration: Since the obfuscation logic is fully exposed in crypt.js and the server accepts any user-defined entropy (rand/rand2), attackers can pre-compute thousands of login payloads per second without interacting with the device until the final injection.
* Credential Recovery via Sniffing: An attacker performing a Man-in-the-Middle (MitM) attack on the local network can intercept the HTTP POST request to login.php. By extracting the last 4 digits of the encrypt parameter, the attacker can instantly reverse the XOR operation and recover the plain-text administrative password.
* Protocol Design Flaw: The server-side authentication fails to provide a server-generated challenge (nonce), trusting the client's self-reported encryption parameters. This violates basic principles of secure authentication (CWE-306).

### Root Cause Analysis:
![](/img/login_crypt_file.png) <br>
![](/img/crypt_js.png) <br>
The function encrypt(passwd) in crypt.js generates two random variables, rand and rand2, to create an opt key for a XOR operation. <br>
![](/img/encrypt_console.png) <br>
However, the final payload is constructed as base64(xor_result) + rand + rand2. <br>
![](/img/pass_py.png) <br>
The backend script login.php retrieves these last 4 digits to "decrypt" the password. Because the server does not validate if these parameters were generated by a trusted source, it allows an attacker to control the decryption logic.

---

### Code Flow:
1. The user enters a password in the UI.
2. mySubmit() calls encrypt(passwd), which applies (charCodeAt ^ (rand >> rand2)).
3. The resulting string is appended with the rand values and placed in the <input type="hidden" name="encrypt"> field.
4. The original password field is cleared, and only the encrypt value is sent to login.php.
5. An attacker can bypass this by using a Python script to generate valid encrypt payloads for any password using a fixed opt key (e.g., rand=031, rand2=4).
---

### Suggested Fix:
* Transition to Industry Standards: Replace the custom XOR obfuscation with a standard TLS/HTTPS implementation to protect data in transit.
* Server-Side Entropy: If HTTPS is not available, implement a proper Challenge-Response mechanism where the rand values are generated by the server and tied to a unique session token (Nonce), preventing pre-computation of hashes.

import base64

def encrypt_raisecom(password, rand_val=31, rand2_val=4):
    rand_str = f"{rand_val:03}"
    rand2_str = str(rand2_val)
    
    opt = rand_val >> rand2_val
    
    encoded_chars = [str(ord(char) ^ opt) for char in password]
    string_para_b64 = ",".join(encoded_chars)
    
    b64_bytes = base64.b64encode(string_para_b64.encode('utf-8'))
